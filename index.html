こんなふうに使い分ける
01

ファイル作成 — スクリプトやドキュメントを作る

GAS・Chrome拡張・Webアプリなど何でも

📞
チャット
まず設計と要件を相談する
「こういう仕組みを作りたい」→ 必要なファイル構成・ロジックを一緒に考える
CC
Claude Code
チャットで決めた内容を丸投げで生成
「このGASスクリプト作って」と指示するだけ。複数ファイルも一気に出してくれる
丸投げ型 — 完成イメージが固まってるならこっち一択
🔧
Cursor（必要なときだけ）
コードを開いて条件分岐の微調整・動作確認
Claude Codeで一発OKなら飛ばしてOK。「ここの判定ロジック変えたい」みたいな細かい修正向き
Point — チャットで設計 → Claude Codeで一気に作成 → 必要ならCursorで微調整。この流れで大体のファイルは完成する。
02

ファイル整理 — フォルダの中身を整える

コード不要、自然言語で指示するだけ

You
あなた
Coworkに指示を出す
「このフォルダの請求書PDFを取引先別にサブフォルダに分けて」
📁
Cowork
ファイル読み取り → 分類 → フォルダ作成 → 移動を全自動実行
PDF読み取り、リネーム、まとめてスプシに転記 — こういう雑務が最も輝く場面
「全部読んで要約して」「日付順に並べ替えて」もOK
ここがすごい — コードが1行も書けなくても使える。「〇〇して」と日本語で言うだけ。ファイル整理の雑務が消える。
03

実行＋AI判断 — 処理の途中でAIに考えさせる

freee会計連携＋フォルダ内ファイル＋AI判断の入力作業

CC
Claude Code（事前準備）
freee API連携スクリプトを作成
ファイル読み取り → 判断ロジック → 入力処理を一気に書く
📁
Cowork（事前準備）
処理対象のファイルをフォルダに整理・リネーム
「請求書PDFを取引先別にまとめて」
⚡
ターミナル版 Claude Code（本番）
スクリプトを実行しながら、AIが1件ずつ判断して入力
ファイルを1件読む → AIが内容を判断 → 結果を入力 → 次のファイルへ
✅ 自信のある判断 → そのまま入力
⚠️ 迷う案件 → 「要確認」フラグをつけて飛ばす
📋 最後に要確認リストを人間に提示
👤
人間（最終確認）
AIが保留にした案件だけ確認して判断する
最初は全件ダブルチェック → 精度が安定したら保留分だけ確認に移行
ここがすごい — 「コード実行」と「AI判断」が同じ場所で一体化しているのはターミナル版Claude Codeだけ。Coworkだと「途中で考える」が苦手。Cursorはコード編集向きで実行向きではない。
04

ブラウザ操作 — APIがないサービスを動かす

freee Signの住所入力など、画面操作が必要なケース

⚡
ターミナル版 Claude Code + --chrome
ターミナルからChrome上のサービスを直接操作
freee Signのフォーム入力、管理画面の操作など
✓
注意
API連携できるならそっちの方が圧倒的に速い
ブラウザ操作 = 最終手段。まずAPIがあるか確認するのが先
ブラウザ操作は遅い＋不安定になりやすいので、できるだけ避ける
優先順位：① API連携 → ② ターミナル+chrome → ③ Claude in Chrome。APIが使えるなら①一択。
⚠️ よくある失敗：全部1つのツールでやろうとすること。
チャットで相談 → Claude Codeで作成 → Coworkで整理 → ターミナルで実行。工程ごとにツールを切り替えるのが最速。
Cursorの中の使い分け — モード × モデル
Cur

Cursor = 2つの選択がある

左側で「何をさせるか」（モード）、右側で「誰にやらせるか」（モデル）を選ぶ

モード = 作業の種類 モデル = AIの頭脳
∞
4つのモード（左側のドロップダウン）
「何をさせるか」を選ぶ
∞
Agent
自律型 — 一番使う
Claude Codeと同じ丸投げモード。ファイル編集・コマンド実行・検索を全部自分で判断して進めてくれる。基本はこれでOK
⊞
Plan
設計図を先に作る
いきなり手を動かさず、まず計画を立ててから実行。複雑なタスクで「何をどの順番でやるか」を確認したいとき
☆
Debug
エラー修正専門
エラーメッセージを貼り付けると原因分析→修正を自動実行。「動かない！」ときはこれ
💬
Ask
質問だけ
ファイルを編集せず質問に答えるだけ。コードの意味を聞きたい、仕組みを教えてほしいとき。チャットに近い
迷ったらAgent一択。大体これで済む。Planは「大きな変更の前に計画を見たい」とき、Debugは「エラーが出た」とき、Askは「触らないで教えて」のときだけ。
🧠
モデル選択（右側のドロップダウン）
「誰にやらせるか」を選ぶ = AIの頭脳を切り替える
モデル特徴おすすめ場面
Auto★ 推奨 自動で最適モデルを選択迷ったらこれ。無制限で使える
Composer 1.5Cursor独自モデル（デフォルト）Cursor内の操作に最適化されてる
Sonnet 4.6Anthropic製。速度と品質のバランス型速さ重視のコーディング。コスパ良い
Opus 4.6Anthropic製。最高品質複雑な設計・難しいバグ・重要な判断
GPT-5.3 CodexOpenAI製。コード生成特化コード生成に特化したい場面
GPT-5.4OpenAI製。汎用・思考力高い複雑な推論が必要な場面
Gemini 3 FlashGoogle製。最速単純作業を大量にこなすとき
MAX
Auto と MAX Mode（トグル）
モデル一覧の上にあるスイッチ
Auto = ON推奨
AIが勝手に最適モデルを選ぶ。無制限で使える。基本はこれだけでOK
MAX Mode = 使用量2〜3倍消費
より多くの計算リソースを使う。難しい問題に本気を出すモード。普段はOFFで十分
Cur

Cursorの使い分け早見表

結局どうすればいいの？

🔧
普段の作業（90%の場面）
Agent + Auto
迷わずこの組み合わせ。自動で最適なモデルが選ばれて、自律的にコードを書いてくれる
無制限で使えるから、気軽にバンバン指示を出してOK
🐛
エラーが出たとき
Debug + Auto
エラーメッセージを貼り付けるだけ。原因を自動分析して修正してくれる
📋
大きな変更の前
Plan + Auto
「10ファイルを一気にリファクタリング」みたいな大規模タスクで、先に計画を確認したいとき
💬
触らないで教えてほしいとき
Ask + Auto
「このコード何やってるの？」「このエラーの意味は？」— ファイルを変更せず質問だけしたいとき
難
本当に難しい問題
Agent + Opus 4.6（または MAX Mode ON）
Autoで解けなかった複雑な問題に、最強モデルを直接指名。使用量は多いが精度が段違い
結論 — 90%は「Agent + Auto」で終わる。残り10%はDebug/Plan/Ask/Opusを場面に応じて。
💰 無料プランの注意：Auto無制限 + プレミアムモデル（Opus等）は月50回まで。
Autoだけで十分強いので、プレミアムは「Autoで解けなかった問題」の切り札として温存するのがコツ。
